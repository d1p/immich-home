# Backup & Recovery

This document covers the backup strategy, how to verify backups, and how to fully recover if your drive fails.

---

## Backup Strategy

| Copy | Where | What | Frequency |
|---|---|---|---|
| **1 — Local** | Your PC (`UPLOAD_LOCATION`) | Originals, library assets, profile images, and DB dumps — all on the same drive | Continuous / Daily at 02:00 |
| **2 — Offsite** | Amazon S3 Glacier Deep Archive | Everything in `UPLOAD_LOCATION`, including DB dumps | Daily at 03:00 (by backup-glacier) |

> **This is not a true 3-2-1 backup.**
>
> A 3-2-1 requires three distinct copies on two different storage media. The Immich DB dumps (`UPLOAD_LOCATION/backups/`) are generated daily by Immich and are useful for metadata recovery, but they live on the **same physical drive** as your originals. A single drive failure takes both out at once — they are one copy, not two.
>
> What you actually have is **2 copies: local HDD + offsite Glacier**. This protects against the most common disasters (drive failure, accidental deletion, ransomware). If you want a true 3-2-1, add a second local copy on a separate drive or NAS.

The offsite backup runs 1 hour after Immich generates its DB dump, so originals and database always travel to Glacier together.

---

## What Is Backed Up

| Path | Contents | Backed up? |
|---|---|---|
| `UPLOAD_LOCATION/upload/` | Original uploaded assets | ✅ Yes — **critical** |
| `UPLOAD_LOCATION/library/` | Assets organised via storage template | ✅ Yes |
| `UPLOAD_LOCATION/profile/` | User profile photos | ✅ Yes |
| `UPLOAD_LOCATION/backups/` | Daily PostgreSQL dumps generated by Immich | ✅ Yes |
| `UPLOAD_LOCATION/thumbs/` | Generated thumbnails | ❌ No — regenerable |
| `UPLOAD_LOCATION/encoded-video/` | Re-encoded video versions | ❌ No — regenerable |

Thumbnails and encoded videos are excluded because they can be regenerated from originals, and excluding them saves significant storage cost.

## Backup Service: `backup-glacier`

The `backup-glacier` Docker service runs `rclone sync` to Amazon S3 Glacier Deep Archive.

### Schedule

- **Daily at 03:00** — runs automatically via the internal cron loop in `scripts/backup-entrypoint.sh`
- **Startup catch-up** — if the container starts and the last sync was more than 24 hours ago, it syncs immediately without waiting for 03:00

### Triggering a manual backup

```powershell
docker compose run --rm backup-glacier
```

Logs are written to `./logs/glacier-<timestamp>.log`.

### Viewing backup logs

```powershell
# Most recent log
Get-ChildItem logs\glacier-*.log | Sort-Object LastWriteTime -Descending | Select-Object -First 1 | Get-Content

# Live log from the running service
docker compose logs backup-glacier --follow
```

### Verifying the last sync time

The service writes a Unix timestamp to `./logs/.last-sync-success` after every successful run.

```powershell
# Human-readable last sync time
$ts = Get-Content logs\.last-sync-success
[DateTimeOffset]::FromUnixTimeSeconds($ts).LocalDateTime
```

## Setting Up Offsite Backups (First Time)

1. Create an AWS account and S3 bucket — see [aws-glacier-setup.md](aws-glacier-setup.md) (step-by-step guide, no AWS experience needed)
2. Fill in your `.env`:
   ```
   AWS_ACCESS_KEY_ID=your-key-id
   AWS_SECRET_ACCESS_KEY=your-secret-key
   S3_BUCKET_NAME=your-bucket-name
   S3_REGION=us-east-1
   ```
3. Restart the compose stack: `docker compose up -d`
4. Trigger a first manual backup to verify everything works:
   ```powershell
   docker compose run --rm backup-glacier
   ```

## Disaster Recovery

> **Important**: Glacier Deep Archive retrieval takes **12–48 hours**. You cannot download files instantly. Plan ahead — initiate the restore request as soon as you know you need it.

### Option A — Restore from local DB backup (drive is OK, DB corrupted)

If your drive is fine but the database is corrupted or you made a mistake:

1. Open Immich: **Administration → Maintenance → Restore**
2. Select the most recent `.sql.gz` file from the list
3. Immich will restore and restart automatically

---

### Option B — Full restore from Glacier (drive failure / data loss)

This uses `scripts/restore-from-glacier.ps1`. You need:
- [AWS CLI](https://aws.amazon.com/cli/) installed
- Your bucket name and region

#### Step 1 — Initiate the thaw

This tells AWS to start restoring files from Glacier storage to a temporary S3 location.
The thaw takes 12–48 hours.

```powershell
.\scripts\restore-from-glacier.ps1 -Initiate -BucketName "your-bucket-name" -Region "us-east-1"
```

Optional parameters:
| Parameter | Default | Description |
|---|---|---|
| `-RestoreDays` | `3` | Days the thawed copy stays available for download (1–30). Give yourself enough time to download. |
| `-Prefix` | `immich` | S3 key prefix. Only restore the folder you need (e.g., `immich/upload`). |

#### Step 2 — Check restore status

Run this any time to see how many objects are thawed and ready:

```powershell
.\scripts\restore-from-glacier.ps1 -Status -BucketName "your-bucket-name"
```

#### Step 3 — Download thawed data (12–48 hours after Step 1)

```powershell
.\scripts\restore-from-glacier.ps1 -Download -BucketName "your-bucket-name" -LocalPath "D:\immich-restore"
```

This uses `aws s3 sync` to download everything to `D:\immich-restore`.

#### Step 4 — Rebuild Immich

1. Stop Immich: `docker compose down`
2. Move the restored files to your `UPLOAD_LOCATION` path
3. Start Immich: `docker compose up -d`
4. Restore the database: **Administration → Maintenance → Restore** → pick the latest `.sql.gz` from `UPLOAD_LOCATION/backups/`
5. Rescan the library: **Administration → Jobs → Library → Scan All**

Immich will re-generate thumbnails and encoded videos automatically. This takes time depending on library size but requires no user action.

## Cost Estimates

| Library size | Storage/month | Annual |
|---|---|---|
| 100 GB | ~$0.10 | ~$1.20 |
| 200 GB | ~$0.20 | ~$2.40 |
| 500 GB | ~$0.50 | ~$6.00 |
| 1 TB | ~$1.00 | ~$12.00 |
